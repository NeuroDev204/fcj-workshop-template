This section verifies the AWS services used by the workout generation flow. The goal is to confirm that the request goes from the frontend to API Gateway, reaches Lambda, runs the Rule Engine, stores the result in DynamoDB, and writes execution logs to CloudWatch.

#### Step 1 - Verify Lambda Rule Engine

Open **AWS Console > Lambda > Functions > `periodiq-rule-engine-dev`**.

Confirm the following information:

- Function name is `periodiq-rule-engine-dev`
- Description mentions the workout generation route `/api/workoutplans/generate`
- API Gateway is attached as the trigger
- Runtime is Amazon Linux 2023
- The function has a recent `Last modified` timestamp

![Lambda Rule Engine detail](../../../images/5-Workshop/5.4-Nguoi2-RuleEngine/01-lambda-rule-engine-detail.png)

This Lambda is the compute unit responsible for running the backend application path that generates a plan. In this project, the Rule Engine applies three groups of rules:

| Rule stage | Purpose |
|---|---|
| **Volume Filter** | Controls training volume based on goal, level, and days per week |
| **Conflict Resolution** | Reduces stress when limitations or CNS-related conflicts appear |
| **Progression Builder** | Builds week-to-week progression and creates a deload week |

The expected 4-week structure is:

```text
Week 1: Build base volume
Week 2: Volume overload
Week 3: Peak intensity
Week 4: Deload week
```

#### Step 2 - Verify API Gateway route

Open **AWS Console > API Gateway > `periodiq-dev` > Routes**.

Select the route:

```text
ANY /api/workoutplans/generate
```

![API Gateway workout generation route](../../../images/5-Workshop/5.4-Nguoi2-RuleEngine/02-api-gateway-workout-generate-route.png)

This route is the public API entry point used by the frontend when the user clicks **Generate 4-Week Plan**. The request body contains the user training parameters:

```json
{
  "goal": "Hypertrophy",
  "fitnessLevel": "Intermediate",
  "daysPerWeek": 4,
  "startDate": "2026-07-06",
  "mainLifts": {
    "squat": 100,
    "bench": 80,
    "deadlift": 120
  }
}
```

The API route forwards the request to Lambda. Lambda then calls the service layer to generate and save the workout plan.

#### Step 3 - Verify Amazon S3 frontend hosting

Open **AWS Console > S3 > `periodiq-frontend-dev-345798130457` > Objects**.

Confirm that the deployed React + Vite build files exist:

- `index.html`
- `assets/`
- `favicon.svg`
- `icons.svg`

![S3 frontend files](../../../images/5-Workshop/5.4-Nguoi2-RuleEngine/03-s3-frontend-files.png)

S3 stores the static frontend files. The Workout Plan screen is part of this SPA and calls the backend API through the configured API base URL.

#### Step 4 - Verify generated plans in DynamoDB

Open **AWS Console > DynamoDB > Explore items > `WorkoutPlan`**.

Run a scan and verify that generated plans exist.

![WorkoutPlan items in DynamoDB](../../../images/5-Workshop/5.4-Nguoi2-RuleEngine/04-dynamodb-workoutplan-items.png)

Important fields to check:

| Field | Meaning |
|---|---|
| `Id` | Unique plan ID |
| `UserId` | User who generated the plan |
| `Goal` | Training goal such as Hypertrophy or Endurance |
| `FitnessLevel` | Beginner, Intermediate, or Advanced |
| `StartDate` / `EndDate` | 4-week program date range |
| `Status` | Current plan status |
| `Weeks` | Nested weekly program generated by the Rule Engine |

The `Weeks` field is the main output of my feature. It contains the week, day, and exercise structure returned to the frontend.

#### Step 5 - Verify CloudWatch log groups

Open **AWS Console > CloudWatch > Logs > Log groups**.

Confirm that Lambda log groups exist for the backend application.

![CloudWatch log groups](../../../images/5-Workshop/5.4-Nguoi2-RuleEngine/05-cloudwatch-log-groups.png)

For this role, the important log groups are:

```text
/aws/lambda/periodiq-api-dev
/aws/lambda/periodiq-rule-engine-dev
```

These log groups are used to debug API requests, Lambda execution, and workout generation failures.

#### Step 6 - Verify Rule Engine execution log

Open the latest log stream inside:

```text
/aws/lambda/periodiq-rule-engine-dev
```

Generate a plan from the frontend, then confirm the log stream contains:

- `POST /api/workoutplans/generate`
- HTTP status `200`
- Lambda `START`, `END`, and `REPORT`
- SQS message after plan generation

![CloudWatch Rule Engine generate log](../../../images/5-Workshop/5.4-Nguoi2-RuleEngine/06-cloudwatch-rule-engine-generate-log.png)

This proves that the workout generation endpoint executed on AWS, returned successfully, and continued the asynchronous flow by sending a message to SQS.

#### Result

At this point, the AWS side of my role is verified:

```text
Frontend on S3
  -> API Gateway route /api/workoutplans/generate
  -> Lambda Rule Engine
  -> DynamoDB WorkoutPlan item
  -> CloudWatch execution log
```

No cleanup was performed because these resources are shared by the PeriodIQ team and are required for the final demo.
